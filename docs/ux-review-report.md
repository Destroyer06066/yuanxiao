# 院校管理平台 UX 体验评审报告

> 评审日期：2026-03-29
> 评审团队：PM-A（运营管理员视角）、PM-B（院校管理员视角）、PM-C（院校工作人员视角）
> 评审方法：独立体验 → 交叉验证 → 综合汇总

---

## Executive Summary

本次体验评审由三位 PM 分别代表 **运营管理员（OP_ADMIN）**、**院校管理员（SCHOOL_ADMIN）**、**院校工作人员（SCHOOL_STAFF）** 三个角色，对系统进行了全链路体验评审。三位 PM 共发现 **23个问题**（其中 P0 严重Bug 4个，P1 高优先级 9个，P2 中优先级 7个，P3 低优先级 3个）。

**总体评价**：系统架构清晰，状态机设计合理，权限体系完善，UI 基于 Element Plus 整体规范。但存在若干影响核心业务操作的体验断点和数据展示缺陷，需分阶段优化。

**最高优先级修复项**：
1. 忘记密码链接404（影响首次登录用户）
2. 专业排名显示 UUID 而非专业名称（影响所有角色的核心统计）
3. 名额编辑无业务校验（存在数据一致性风险）
4. 批量拒绝无操作人校验（可能误操作）

## 一、项目背景

### 1.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | Vue 3 + TypeScript + Vite |
| UI 组件库 | Element Plus 2.7.0 |
| 状态管理 | Pinia |
| 后端框架 | Spring Boot 3.4.0 + Java 21 |
| 数据库 | PostgreSQL + MyBatis-Plus |
| 认证方式 | JWT |
| 图表 | ECharts 5.5.0 |

### 1.2 用户角色体系

| 角色 | 角色 Key | 核心职责 |
|------|----------|----------|
| 运营管理员 | OP_ADMIN | 管理所有院校和账号、系统配置、跨院校数据监控 |
| 院校管理员 | SCHOOL_ADMIN | 管理本校考生录取决策、名额管控、专业配置、数据统计 |
| 院校工作人员 | SCHOOL_STAFF | 查看本校数据，进行辅助性操作（核验、报到） |

### 1.3 核心功能模块

- **招生录取管理**：考生推送 → 直接录取/有条件录取 → 终裁 → 确认 → 材料收件 → 报到
- **名额管理**：专业名额配置、名额使用跟踪、名额预警（超90%）
- **报到管理**：材料收件登记、报到确认
- **成绩核验**：成绩证书验证、验证历史
- **补录管理**：补录轮次配置、轮次内操作
- **数据统计**：KPI 指标、月度趋势、院校排行、预警
- **系统管理**：院校管理、账号管理、角色权限、审计日志

### 1.4 状态机

```
PENDING → 直接录取 → ADMITTED → CONFIRMED → MATERIAL_RECEIVED → CHECKED_IN
PENDING → 有条件录取 → CONDITIONAL → 终裁 → ADMITTED
PENDING → REJECTED（终态）
* → INVALIDATED（终态，被其他院校确认时）
```
